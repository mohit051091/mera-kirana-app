# Standard Operating Procedures (SOP): Mera Kirana E-Commerce Suite

This document serves as the official handbook for operating, maintaining, and developing the Mera Kirana WhatsApp order chatbot and admin dashboard.

---

## SOP 1: The Business Owner's Operations Guide

### 1. Accessing the Config Panel
1. Log in to the Admin Dashboard using your credentials.
2. Navigate to the **Settings** tab in the sidebar menu.
3. To update **Serviceable Areas**:
   - Scroll to "Serviceable Pincodes".
   - You can search by City/District (e.g. "Mumbai") or Taluk (e.g. "Bhandup") and add them in bulk.
   - You can also upload a one-column Excel/CSV file with a list of pincodes.
   - Selected areas will be displayed as pill tags. Click the `x` button on a tag to delete it.
4. To update **Operating Hours & Slot Limits**:
   - Set day-wise operating hours.
   - Set **Rider Booking Limits** per delivery slot (e.g. max 10 orders per slot). Once a slot reaches capacity, the bot automatically hides it from customer checkout.
   - Customers can place orders 24/7; they will be shown next-day/next available slots.
5. To toggle **Vacation Mode**:
   - Toggle the "Shop Status" switch to **Closed**. This automatically redirects all incoming WhatsApp messages to a friendly closure auto-reply.

### 2. Remarketing Constructor & Predictive Scheduling
1. Open the **Remarketing Configs** panel.
2. Configure **ON/OFF** status for cart recovery messaging.
3. Set the **Sequence Builder**: Specify the number of recovery attempts (e.g., Max 3) and custom interval delays (e.g., 30 mins, 2 hours, 4 hours).
4. Select the target stages for campaigns (e.g. "Run recovery strictly on payment page drop-offs").
5. Toggle **Intelligent Scheduling (Predictive Delivery)**:
   - When enabled, the server matches the variant (e.g., Paneer) to its historical order peak hours and schedules the recovery text right before this window.
6. Set A/B Testing: Split Group A (receives discount dynamically filled in pre-approved Meta variable template) vs. Group B (receives default recovery template with no discount).
7. **DND Compliance:** All campaign messages and automated sequences feature a `[🔕 Stop Messages]` opt-out button. When clicked, the bot disables future promotional messages to the number to prevent Meta account suspensions.

### 3. Sourcing Forecasting & EOD Performance PDF
1. Go to the **Sourcing** tab on the sidebar.
2. Review the weekly predictive sourcing sheet. The sheet displays historical sales baselines and adjusts procurement suggested lists using external weather patterns (Open-Meteo) and holiday/festive calendars.
3. **Consolidated EOD PDF Report:** Every night at 10:00 PM, the system compiles a dynamic PDF report summarizing daily, weekly, and monthly net sales, profit margins, conversion drop-off percentages, and active campaign yields. This PDF is automatically sent to your personal WhatsApp number.

### 4. Managing Coupons & Festive Campaigns
1. Go to the **Sales & Coupons** tab in the sidebar.
2. Click **Create Coupon**.
3. Input the parameters:
   - **Code:** e.g., `FESTIVE20` (or click "Auto-Generate").
   - **Type:** Percentage or Flat rate.
   - **Value:** e.g., `15` for 15% off.
   - **Limits:** Set a date range and input the **Usage Cap** (e.g., `100` for the first 100 users).
4. Review the Campaign Analytics table below to track: *Total Orders Redeemed*, *Discounts Granted*, and *Net Sales Revenue* generated by the campaign.

### 5. Salesperson Incentives & Commissions
1. Go to the **Referral Partners** tab.
2. To add a new agent, click **Register Agent**.
3. Enter their Name and Phone number. The system will auto-generate a code using the format: `[First 3 letters of Name] + [Last 3 digits of Phone] + [DDMMYY]` (e.g. `RAJ450170726`).
4. Set the **Incentive Rate** (e.g., 5% of order value, or flat ₹20 per checkout).
5. The dashboard ledger displays their referred customer counts, successful orders, and accumulated commissions.

### 6. Monitoring Revenue & Margins (Owner-Only)
1. Profit margin metrics, cost prices, and revenue analytics are **restricted strictly to the Owner role**. Admin and Logistics managers will not see CP fields or margins.
2. On the **Orders** page, view the margin breakdown next to each order.
3. On the **Catalog** tab, configure the **Cost Price (CP)** of each item variant.
4. The main dashboard home screen dynamically updates:
   - **Total Net Profit** = Selling Price - Cost Price (cumulative).

## SOP 2: The Delivery Partner's (Riders) Guide

### 1. Delivery Route Optimization
1. Log in to the Rider mobile web app.
2. Select your assigned morning/noon delivery batch.
3. The app displays orders **pre-sorted in the optimal sequence** based on the Open Source Routing Machine (OSRM) traveling salesperson calculations (minimizing travel time and fuel consumption).
4. Tap the first order in the list, then tap **"Navigate"** to open turn-by-turn navigation in Google Maps.

### 2. Verifying Delivery: Scenario A (Hand-off Delivery)
1. Arrive at the customer's location.
2. Ask the customer for the **4-digit Delivery OTP** sent to their WhatsApp.
3. Input the OTP in the rider app and tap **Verify**. The order status updates to `DELIVERED` and triggers invoice receipts.

### 3. Verifying Delivery: Scenario B (Unattended Door Drop)
1. Use this method **only** if the customer is sleeping (early morning drops) or unavailable.
2. Hang/place the bag securely on the door handle or gate.
3. Open the Mobile Rider Web App and toggle **"Drop at Door"**.
4. Take a photo showing both the package and the door number.
5. Tap **Upload**. (The system will check your GPS coordinates to verify you are at the correct location, then complete the delivery).

## SOP 3: Customer Service & Dispute Resolution Guide

### 1. Handling "Report Missing" Delivery Disputes
1. If a customer clicks **"Report Missing"** on WhatsApp after an unattended drop:
   - A red alert triggers on the Admin Dashboard: **“Disputed Delivery - Order #1042!”**
2. Open the order detail page. You will see:
   - The door photo uploaded by the rider.
   - The GPS timestamp match.
3. **Action Flow:**
   - Call the delivery rider immediately (who is still in the local neighborhood).
   - Call the customer to verify if the bag was picked up by a family member.
   - If verified as stolen/rider fraud, initiate the refund process.

### 2. Approving Cancellations & Issuing Refunds
1. When a cancellation request arrives in the dashboard cancellation queue:
2. Review the order. If it has not been dispatched, click **Approve Cancellation**.
3. **Exception Handling & Return Window:** Customers can request a refund on WhatsApp up to **3 hours after delivery**. These land in the Exception Approval Queue.
4. **COD Order Limitation:** Refunds and cancellations are **disabled** for COD checkouts once delivered. The checkout screen shows: *"No cancellations or refunds are permitted on COD orders once hand-off is complete."*
5. **Refunding Online UPI Payments:**
   - Online payments are processed directly to your VPA.
   - Open your bank/UPI app, transfer the refund amount back to the customer's UPI ID, and obtain the transaction UTR number.
   - Enter the **Refund UTR** into the Admin Panel to mark the order as `REFUNDED`.
   - *The bot will automatically send a PDF Credit Note/Refund receipt to the customer on WhatsApp.*

## SOP 4: The Developer's Technical Reference

### 1. Database Mappings & Models
- `system_settings`: Stores config settings (operating hours, serviceable pincodes, fees, slot limits, remarketing schedules, active A/B cohorts).
- `pincode_master`: Indian government pincode database.
- `dropoffs`: Funnel analytics tracking.
- `campaigns`: Marketing metrics ledger (revenue, meta API charges, margin analysis).
- `coupons`: Voucher rates and usage caps.
- `salespeople`: Referral mapping codes and commission calculations.
- `subscriptions`: Recurring delivery schedules.

### 2. API Endpoints Registry
- `POST /api/webhook/whatsapp`: Meta webhook entry point. Parses text and media messages.
- `GET /api/webhook/whatsapp`: Verify token challenge.
- `GET /api/orders` & `PUT /api/orders/:id/status`: Order tracking APIs.
- `POST /api/coupons`: Voucher generator routes.
- `PUT /api/catalog/sync`: Pushes dashboard changes to Meta Commerce Manager.
- `POST /api/payments/confirm`: Webhook confirm from Flat-Rate UPI gateway.
- `POST /api/campaigns/broadcast`: Schedules template-based marketing/remarketing campaigns.

### 3. Multi-Stage Conversational Voice Processing (Gemini 1.5 Flash SDK)
- The backend utilizes the `@google/generative-ai` package to send `.ogg` audio files directly to Google AI Studio hosted endpoints (avoiding GPU costs on Railway).
- **State Flow Extraction:**
  - **Cart State:** Gemini extracts items and counts (matching synonym dictionaries: e.g. Khoya = Mawa).
  - **Address State:** Gemini extracts the location variables (`street`, `building`, `area`, `landmark`, `pincode`) from their spoken address note.
  - **Slot State:** Gemini parses time expressions (e.g. "kal dopahar tak") and matches them to slots.
  - **One-Shot Checkout Parsing:** If items, address, and slot are spoken together, Gemini populates all JSON properties at once. The server immediately bypasses checkout states and issues the invoice payment link.

### 4. Extreme AI Sandbox Isolation & Security Protocol (Minutes of Governance)
To prevent prompt injections and system tampering, developers must enforce strict isolation boundaries:
1.  **Zero-Privilege LLM Execution:**
    - The Gemini API key has zero read/write permissions to database resources or system files. 
    - The LLM has **no direct access** to backend application APIs, host servers, network directories, or configuration passwords.
2.  **Strict Structured JSON Wall:**
    - Gemini's response schema is enforced statically. Generative text generation is completely disabled.
    - If Gemini attempts to generate off-topic responses or injection payloads (e.g. system files requests), the API fails the schema enforcement and outputs empty fields.
3.  **Backend Privilege Control:**
    - The backend does **not** execute system commands, execute SQL queries, or trigger destructive functions based on LLM outputs.
    - Gemini's output is treated solely as unverified raw data. The Node.js server acts as the absolute gatekeeper, validating data types programmatically before modifying database tables.
4.  **No Direct LLM-to-Customer Output:**
    - Gemini never generates conversational text replies to the customer. 
    - All outgoing messages are statically mapped template texts stored in the secure Node.js backend. If Gemini outputs are invalid, the backend sends a standard, safe pre-written template.

### 5. Dynamic Route Optimization (OSRM TSP Solver)
- Backend endpoint `GET /api/partners/route/:rider_id` queries all orders assigned to a rider for the current slot.
- It builds a coordinate array and requests distances from OSRM: `http://router.project-osrm.org/table/v1/driving/{coordinates}`.
- The server runs a Traveling Salesperson nearest-neighbor solver on the matrix and returns the sorted order array to the rider web view.

### 6. Pincode CSV Seeding
- Run the script `node server/database/seed_pincodes.js` during server deployment. The script reads the user's `pincodes.csv` file and batch-inserts all India postal mappings.
